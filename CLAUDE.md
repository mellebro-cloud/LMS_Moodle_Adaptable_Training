# Layer 2: Course Player / Inside Selected Course Architectural Definition

## Purpose

Layer 2 is the selected-course learner experience. It begins after a learner clicks **Enter**, **Continue**, or **Go to Course** from the learner dashboard, My Classroom page, catalog enrollment result, or course card.

Layer 2 must provide one complete ed2go-style course player experience for the selected Moodle course.

For the main template course, the master learner URL is:

`http://localhost/moodle/local/heyday_courseplayer/index.php?id=105`

The master component is:

`local_heyday_courseplayer`

The master route pattern is:

`local_heyday_courseplayer/index.php?id=COURSEID&page=PAGE&cmid=CMID&subpage=SUBPAGE`

---

## Core Rule

There must be:

* one selected-course learner shell
* one left sidebar
* one routing system
* one completion/lock/release-date system
* one Next Up system
* one selected-course visual language
* one Moodle-clutter reduction strategy

That shell is owned by:

`local_heyday_courseplayer`

Moodle core remains the real learning engine.

---

## Layer 2 Master Component

`local_heyday_courseplayer` is the only full learner shell for a selected course.

It owns:

* black or near-black top player bar
* sticky white left sidebar
* light gray page background
* centered white content card
* active sidebar state
* green completion checks
* blue in-progress indicators
* lock icons
* release-date text
* Next Up flow
* page routing
* Moodle clutter reduction inside the player
* consistent ed2go-style page layout

The courseplayer must display and route all selected-course learner pages:

* Home
* Scores
* Discussions
* Learning Objectives
* Getting Started
* Pretest
* Lessons
* Lesson Discussions
* Lesson Assignments
* Lesson Quizzes
* Resources
* Final Exam

---

## Moodle Core Responsibility

The courseplayer must not replace Moodle core functionality.

Moodle remains the source of truth for:

* course structure
* sections
* subsections
* activities
* pages
* quizzes
* question attempts
* forums
* forum posts
* assignments
* files
* folders
* H5P activities
* completion tracking
* availability restrictions
* grades
* quiz attempts
* assignment submissions
* forum permissions
* user permissions
* group restrictions
* access rules

The courseplayer may restyle and wrap Moodle content, but it must not break Moodle’s real activity behavior.

---

## Adaptable Theme Responsibility

Adaptable is responsible only for the global Moodle shell:

* site header
* logo
* global colors
* fonts
* footer
* general Moodle appearance
* site-wide button style
* public learner-facing theme consistency

Adaptable must not rebuild the lesson player, sidebar, course sequence, quiz layout, discussion system, or learner routing.

The courseplayer owns the selected-course experience after the learner enters a course.

---

## Course Sequence

The courseplayer reads the real Moodle course structure from the selected course and converts it into the ed2go-style learner sequence:

Home
→ Scores
→ Discussions
→ Learning Objectives
→ Getting Started
→ Pretest
→ Lesson 1
→ Lesson 2
→ Lesson 3
→ ...
→ Resources
→ Final Exam

For the main template course, the selected course is:

Course ID: `105`

The courseplayer must use Moodle course structure, activities, completion data, availability data, and grade data to build the learner sequence.

---

## Sidebar Requirements

The left sidebar must be generated by `local_heyday_courseplayer`.

The sidebar must show:

* Home
* Scores
* Discussions
* Learning Objectives
* Getting Started
* Pretest
* Lesson groups
* Resources
* Final Exam

Lesson groups must expand and collapse.

Each lesson group may contain:

* Lesson Introduction
* Learning Objectives
* Chapter pages
* Reading pages
* Learning checks
* Lesson Review
* Lesson Assignment
* Lesson Discussion Area
* Lesson Quiz
* Resources for Further Learning

Sidebar behavior:

* active item has a blue left indicator or arrow
* completed items show green checkmarks
* in-progress items show blue dots or progress indicators
* locked items show lock icons
* locked items remain visible but muted
* locked items are not clickable unless Moodle says they are available
* locked items must not look active
* locked items must not show completed styling
* release-date text appears when Moodle availability restrictions provide a date
* long titles wrap cleanly
* sidebar scrolls independently
* content area must not jump or flicker

Example locked release-date text:

`Lesson 9: Cryptocurrencies will be available on Jun 10, 2026 10:00 AM GMT+3`

`Final Exam will be available on Jun 19, 2026 10:00 AM GMT+3`

It should show:

* course fullname
* course shortname or section code
* course banner image when available
* course completion circle
* score circle
* next incomplete activity
* Continue button
* clean white card layout
* no duplicate Moodle course index
* no duplicate blocks or drawer clutter

Suggested route:

`index.php?id=105&page=home`

---

## Scores Page

The Scores page must be rendered inside the courseplayer shell.

It should use Moodle gradebook data as the source of truth.

It should show:

* ed2go-style list/card rows
* activity name
* type
* status
* grade or score on the right
* locked or unavailable items muted
* search/filter toolbar if available
* download button if available
* clean selected-course layout

Suggested route:

`index.php?id=105&page=scores`

---

## Discussions Page

The Discussions page must be rendered inside the courseplayer shell.

It should use Moodle forums as the source of truth.

The discussion feature module may be implemented by:

`local_heyday_discussions`

But it must not create a separate full learner shell.

`local_heyday_discussions` should act as a discussion/forum feature module called by `local_heyday_courseplayer`.

The Discussions page must show:

* all course-level and lesson-level discussions
* one clean row/card per discussion
* discussion title
* lesson association when available
* latest activity metadata when available
* unread/new state when available
* locked state when Moodle availability restricts access
* closed discussion state when Moodle says new posts are closed
* muted styling for locked or closed discussions
* no duplicate forum rows
* no duplicate Moodle forum table clutter

The Discussions page should support ed2go-style tabs such as:

* All
* Mine
* New
* Bookmarks

These tabs should be implemented progressively and must not break Moodle forum permissions.

A closed discussion should display a clear message such as:

`This discussion has been closed to new posts by learners.`

Learners may still view posts when Moodle permissions allow viewing.

Suggested routes:

`index.php?id=105&page=discussions`

`index.php?id=105&page=discussion&cmid=CMID`

---

## Learning Objectives Page

The Learning Objectives page must be shown inside the courseplayer shell.

It may be:

* a course-level Moodle Page activity
* a generated summary from lesson objective pages
* a helper-rendered content page

The source should remain Moodle course content wherever possible.

Suggested route:

`index.php?id=105&page=objectives`

If the content is a Moodle Page activity:

`index.php?id=105&page=lesson&cmid=CMID`

## Getting Started

Getting Started must be displayed inside the same courseplayer shell.

It must not use a nested inner-card layout that duplicates the player design.

Getting Started may include:

* Course Overview
* Syllabus
* Navigating this Course

Suggested routes:

`index.php?id=105&page=gettingstarted&subpage=overview`

`index.php?id=105&page=gettingstarted&subpage=syllabus`

`index.php?id=105&page=gettingstarted&subpage=navigating`

Getting Started should show:

* centered page title
* action icons
* completion status
* divider
* clean content area
* Next Up card

Implementation warning:

Do not call:

`local_heyday_courseplayer_gettingstarted_definitions($course, $context, $lessongroups)`

before `$lessongroups` is created by:

`$lessongroups = local_heyday_courseplayer_collect_lesson_groups($modinfo, $sections, $course, $context);`

Getting Started completion code must first confirm `$lessongroups` exists and is an array.

## Lesson Discussions

Each lesson may contain its own Moodle Forum activity.

Lesson discussions should appear in two places:

1. inside the lesson group in the left sidebar
2. as linked rows inside the main Discussions page

The same Moodle forum activity must not be duplicated as separate fake content.

The courseplayer should route the forum into an ed2go-style discussion view while preserving Moodle forum functionality.

Suggested route:

`index.php?id=105&page=discussion&cmid=CMID`

The discussion view should support:

* viewing discussion threads
* viewing closed discussion messages
* posting only when Moodle allows posting
* replying only when Moodle allows replying
* respecting groups and permissions
* preserving subscriptions when available
* preserving unread/new indicators when available

---

Assignments must use Moodle Assignment as the source of truth.

The courseplayer may style the assignment landing and submission area, but it must preserve:

* submission status
* due dates
* cut-off dates
* grading status
* files
* online text
* feedback
* permissions

Suggested route:

`index.php?id=105&page=assignment&cmid=CMID`

## Lesson Quizzes

Lesson quizzes must use Moodle Quiz as the source of truth.

The courseplayer may style the quiz landing, question area, review area, and completion display, but it must preserve Moodle quiz functionality.

Suggested route:

`index.php?id=105&page=quiz&cmid=CMID`

`local_heyday_quiz` may act as the quiz bridge/renderer used by the courseplayer.

`local_heyday_quizskin` should be merged into the quiz bridge, retired, or reduced to a CSS-only helper.

---

## Resources

Resources must be displayed inside the courseplayer shell after Lessons and before Final Exam.

Resources may include:

* files
* folders
* URLs
* pages
* H5P activities
* downloadable documents
* external references

Suggested route:

`index.php?id=105&page=resources`

Resource items should remain controlled by Moodle availability and permissions.

---

## Final Exam

Final Exam must appear after Resources.

Moodle Quiz remains the real exam engine.

The courseplayer must show:

* Final Exam menu item
* locked state when unavailable
* release-date text when available from Moodle restrictions
* clean exam instruction card
* exam attempt button when available
* grade/pass status when completed
* Next Up or completion notice after submission

Suggested route:

`index.php?id=105&page=finalexam&cmid=CMID`

Locked Final Exam must remain visible but muted.

It must not look active or completed before it is available or passed.

---

## Related Plugin Roles

Older standalone local plugins must not remain independent full learner shells.

Their useful logic should be moved into, called by, or redirected into `local_heyday_courseplayer`.

Final plugin role map:

`local_heyday_courseplayer`
= one selected-course learner shell, router, sidebar, completion, availability, release-date, Next Up, and renderer coordinator

`local_heyday_coursehome`
= Home feature/helper only

`local_heyday_scores`
= Scores feature/helper only

`local_heyday_discussions`
= Discussion/forum feature module only, called by courseplayer

`local_heyday_gettingstarted`
= Getting Started content/helper only

`local_heyday_pretest`
= Pretest helper or legacy redirect only

`local_heyday_lessons`
= lesson structure/service logic only

`local_heyday_quiz`
= quiz bridge/renderer used by courseplayer

`local_heyday_quizskin`
= merge into quiz bridge, retire, or reduce to CSS-only helper

`local_heyday_questionbank`
= admin/course-authoring helper, not learner shell

`local_heyday_helptour`
= support/tour utility, not learner shell

---

## Legacy Plugin URL Rule

Legacy plugin URLs should not keep separate player shells.

They should either:

1. redirect into the courseplayer route, or
2. expose helper functions/classes used by the courseplayer, or
3. remain admin/course-authoring utilities only.

Examples:

`local_heyday_scores/index.php?id=105`

should redirect to:

`local_heyday_courseplayer/index.php?id=105&page=scores`

`local_heyday_discussions/index.php?id=105`

should redirect to:

`local_heyday_courseplayer/index.php?id=105&page=discussions`

`local_heyday_pretest/index.php?id=105`

should redirect to:

`local_heyday_courseplayer/index.php?id=105&page=pretest`

---

## CSS Scope Rule

All courseplayer CSS must be scoped to:

`body.local-heyday-courseplayer`

or:

`.local-heyday-courseplayer`

Do not write broad CSS that affects:

* Moodle admin pages
* normal course pages
* Adaptable settings
* dashboard
* login page
* quiz pages outside the player
* forum pages outside the player
* site-wide Moodle layout

---

## Moodle Clutter Reduction

Inside the courseplayer only, reduce or hide unnecessary Moodle learner clutter such as:

* duplicate breadcrumbs
* duplicate course index
* duplicate blocks
* drawer clutter
* secondary navigation clutter
* duplicate Help
* duplicate Tour
* duplicate Search
* unnecessary forum administration tabs
* unnecessary activity footer navigation when replaced by Next Up

Do not remove Moodle functionality required for quizzes, forums, assignments, grading, attempts, submissions, or permissions.
